# Sitemap详解 - 给非技术人员

## 什么是Sitemap？

### 简单比喻

想象你开了一家大型图书馆：

- **前台页面** = 图书馆里的每一本书（用户来看书）
- **Sitemap** = 图书馆的目录册（告诉管理员有哪些书、在哪里）

**用户**不需要看目录册，他们直接去找书。
**搜索引擎**（Google、Bing）需要目录册，才知道你有哪些书。

---

## 实际例子

### 你的网站结构

```
前台页面（用户访问）：
├── http://47.110.87.81:3000/en              ← 首页
├── http://47.110.87.81:3000/en/papers       ← 论文列表
├── http://47.110.87.81:3000/en/forum        ← 论坛列表
├── http://47.110.87.81:3000/en/forum/1      ← 论坛详情1
├── http://47.110.87.81:3000/en/forum/2      ← 论坛详情2
└── ...更多页面

Sitemap（搜索引擎访问）：
└── http://47.110.87.81:3000/sitemap.xml     ← 包含上面所有URL的列表
```

### Sitemap内容示例

```xml
<urlset>
  <!-- 告诉Google：我有一个首页 -->
  <url>
    <loc>http://47.110.87.81:3000/en</loc>
    <lastmod>2026-03-05</lastmod>
    <priority>1.0</priority>
  </url>

  <!-- 告诉Google：我有一个论文页面 -->
  <url>
    <loc>http://47.110.87.81:3000/en/papers</loc>
    <lastmod>2026-03-05</lastmod>
    <priority>0.9</priority>
  </url>

  <!-- 告诉Google：我有一个论坛详情页 -->
  <url>
    <loc>http://47.110.87.81:3000/en/forum/1</loc>
    <lastmod>2026-03-04</lastmod>
    <priority>0.7</priority>
  </url>
</urlset>
```

## 为什么需要Sitemap？

### 场景1：没有Sitemap

```
你发布了一篇新论文：
http://47.110.87.81:3000/en/forum/100

Google怎么知道这个页面存在？
1. 等待Google爬虫偶然发现（可能要几周）
2. 或者从其他网站的链接找到（如果没人链接，永远找不到）

结果：新内容很久才能被搜索到 ❌
```

### 场景2：有Sitemap

```
你发布了一篇新论文：
http://47.110.87.81:3000/en/forum/100

5分钟后，sitemap自动更新，包含新URL

Google每天访问你的sitemap：
"哦，有新URL了！我去爬取它！"

结果：新内容几小时内就能被搜索到 ✅
```

## Sitemap vs 前台页面

| 特性 | 前台页面 | Sitemap |
|-----|---------|---------|
| **访问者** | 用户（人） | 搜索引擎（机器人） |
| **格式** | HTML（漂亮的网页） | XML（纯文本列表） |
| **内容** | 文章、图片、按钮等 | 只有URL列表 |
| **作用** | 让用户阅读内容 | 告诉搜索引擎有哪些页面 |
| **缓存** | 5分钟（用户体验） | 5分钟（与页面一致） |

## 为什么Sitemap需要缓存？

### 访问频率

```
用户访问前台页面：
- 每天可能几百次
- 每次都是不同的页面

搜索引擎访问sitemap：
- Google每天访问1-3次
- Bing每天访问1-2次
- 其他搜索引擎偶尔访问
- 总计：每天可能10-20次
```

### 如果没有缓存

```
每次访问sitemap都要：
1. 查询papers表（获取所有论文ID）
2. 查询invitation表（获取所有论坛ID）
3. 查询news_column表（获取所有新闻ID）
4. 生成XML文件

每天20次 × 3个查询 = 60次数据库查询
```

### 有缓存后

```
第一次访问：
1. 查询数据库
2. 生成sitemap
3. 缓存5分钟

接下来5分钟内的所有访问：
- 直接返回缓存
- 不查询数据库 ✅

每天最多：288次查询（每5分钟1次）
但实际上可能只有10-20次（因为访问不频繁）
```

## 你之前的担心

### 你说的"前台页面"

你之前说的"如果数据库更新了，就重新刷新前台页面"，指的是：

```typescript
// app/[locale]/papers/page.tsx
export const revalidate = 300; // 5分钟

// 这是前台页面的缓存
// 用户访问 http://47.110.87.81:3000/en/papers 时
// 5分钟内返回缓存的HTML页面
```

### Sitemap的缓存

```typescript
// app/sitemap.ts
export const revalidate = 300; // 5分钟

// 这是sitemap的缓存
// 搜索引擎访问 http://47.110.87.81:3000/sitemap.xml 时
// 5分钟内返回缓存的XML文件
```

**两者是独立的，但应该保持一致！**

---

## 为什么要保持一致？

### 场景：缓存时间不一致

```
假设：
- 前台页面缓存：5分钟
- Sitemap缓存：24小时

你在数据库添加新论文：
00:00 - 添加新论文（ID=100）
00:05 - 前台页面更新，显示新论文 ✅
00:05 - 用户可以访问 /en/forum/100 ✅
24:00 - Sitemap才更新，包含 /en/forum/100 ❌

问题：
- 页面存在了，但sitemap里没有
- Google不知道这个页面存在
- 要等24小时Google才能发现
```

### 场景：缓存时间一致

```
假设：
- 前台页面缓存：5分钟
- Sitemap缓存：5分钟

你在数据库添加新论文：
00:00 - 添加新论文（ID=100）
00:05 - 前台页面更新，显示新论文 ✅
00:05 - Sitemap更新，包含 /en/forum/100 ✅
00:10 - Google访问sitemap，发现新URL ✅
00:15 - Google开始索引新页面 ✅

完美！
```

## 总结

### Sitemap是什么？
- 一个XML文件（不是前台页面）
- 给搜索引擎看的（不是给用户看的）
- 列出网站所有URL（像目录册）

### 为什么需要缓存？
- 避免频繁查询数据库
- 提高响应速度
- 减少服务器负载

### 为什么用5分钟？
- 与前台页面保持一致
- 新内容能快速被搜索引擎发现
- 性能影响很小（查询已优化）

### 你不需要担心
- Sitemap不会影响用户体验（用户看不到）
- 5分钟缓存已经很好地平衡了性能和新鲜度
- 数据库压力很小（每次查询 < 50ms）
